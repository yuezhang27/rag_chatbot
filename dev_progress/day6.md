但这其实是 RAG 系统里一个很容易被低估的环节。让我帮你建立一个清晰的认知框架：

---

**为什么今天"感觉全是提示词"但又很重要？**

你之前的 Day 1–5 建的是**数据管道**（索引、检索、向量化），今天建的是**输出控制层**。

打个比方：前几天你装好了图书馆的书架和检索系统，今天你在训练图书管理员怎么**回答读者的问题**——引用哪本书、不知道时怎么说、回答格式是什么。

这个"训练"的载体，就是 System Prompt。

---

**今天三件事的本质是什么**

**1. 结构化 System Prompt** → 给模型划边界

```
你是 HR 知识库助手
→ 只基于 context 回答（Grounding）
→ 不知道就说不知道（Hallucination 抑制）
→ 回答要带 citation（可追溯性）
```

这不是"随便写几句话"，是在定义模型的**行为契约**。

**2. Chain-of-Thought** → 让模型先想再说

没有 CoT：模型直接生成答案，容易跳跃、编造。

有 CoT：
```
先分析：context 里有哪些相关段落？
再推理：这些段落能回答问题的哪些部分？
最后输出：基于以上，组织回答
```

强迫模型"走流程"，幻觉概率会明显下降，因为它需要先"找到证据"才能说话。

**3. chunk size 手动测试** → 验证检索质量

这个其实不是提示词，是在测试 Day 5 的数据管道。400 tokens / 80 overlap 这个参数对你的 HR 文档合不合适？今天就是用真实问题来感受一下。

---

**引导思考的核心洞察**

> 你改了好几版 prompt，但怎么知道哪版更好？

这就是今天最重要的 takeaway：**Prompt Engineering 是工程，不是玄学**，需要量化指标，这就是 Day 8 RAGAS 的铺垫。

今天你在积累"感性认知"——哪种写法幻觉少、哪种 citation 更准——Day 8 你会把这种感觉变成可测量的数字。
